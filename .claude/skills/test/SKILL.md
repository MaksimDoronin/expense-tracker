Покрой тестами файл по указанному пути: `$0`

## Аргументы скилла

- `$0` — путь к файлу, который нужно покрыть тестами (относительный или абсолютный).

## Шаги

1. Прочитать файл `$0` и понять его назначение: экспортируемые функции, классы, сервисы, компоненты.
2. Определить контекст файла:
   - Путь содержит `apps/backend` → **NestJS/Jest** (unit или e2e).
   - Путь содержит `apps/frontend` → **React Testing Library + Jest**.
   - Путь содержит `packages/shared` → **чистый Jest** (без фреймворков).
3. Найти соседние тестовые файлы рядом (паттерн `*.spec.ts` или `*.test.ts`) — изучить их стиль и соглашения.
4. Проверить, существует ли уже тестовый файл для `$0`:
   - Backend: `<name>.spec.ts` рядом с файлом.
   - Frontend: `<name>.test.tsx` рядом с файлом или в `__tests__/`.
   - Если файл уже существует — дополнить, а не перезаписывать.
5. Составить план тестов: перечислить все публичные функции/методы/компоненты и сценарии (happy path, edge cases, ошибки).
6. Написать тесты и создать/дополнить тестовый файл.
7. Запустить тесты, чтобы убедиться, что они проходят.

## Правила написания тестов

### Backend (NestJS)
- Фреймворк: `@nestjs/testing`, `jest`.
- Мокировать зависимости через `jest.fn()` или `createMock` из `@golevelup/ts-jest`.
- Структура: `describe('<ClassName>')` → `describe('<methodName>')` → `it('<scenario>')`.
- Использовать `Test.createTestingModule` для unit-тестов сервисов.
- Проверять как успешные сценарии, так и ошибки (`rejectWith`, `toThrow`).

### Frontend (React)
- Фреймворк: `@testing-library/react`, `@testing-library/user-event`, `jest`.
- Рендерить через `render()`, находить элементы по `getByRole` / `getByText` / `getByTestId`.
- Тестировать поведение (клики, ввод), а не детали реализации.
- Мокировать fetch/API-вызовы через `jest.mock` или `msw`.
- Асинхронные операции — `waitFor` / `findBy*`.

### Shared (чистые утилиты)
- Только Jest, никаких фреймворков.
- Параметризованные тесты через `it.each` там, где это уместно.

## Соглашения проекта

- Тестовый файл располагается рядом с исходным: `src/auth/auth.service.spec.ts`.
- Импорты — через алиас `@expense-tracker/shared` для общих типов.
- Моки объявляются в начале файла или в `beforeEach`.
- Каждый `describe`-блок — логически завершённая единица.

## Запрещено

- Не писать тесты, которые проверяют детали реализации вместо поведения.
- Не использовать `any` без крайней необходимости.
- Не запускать `npm run build` — только тесты.
- Не коммитить — тесты пишутся, коммит делает пользователь через `/commit`.
